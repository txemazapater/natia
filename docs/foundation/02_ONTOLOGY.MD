# ONTOLOGY

> *"Before defining software, define reality."*

**Status:** Draft  
**Version:** 0.1  
**Last updated:** 2026-07-22

---

# Purpose

This document defines the ontology of NATIA.

Ontology describes the nature of the concepts that exist inside the NATIA universe.

It answers a simple question:

> **What kinds of things exist?**

The ontology is independent from implementation.

It precedes architecture.

It precedes software.

It even precedes NATIA itself.

---

# Why an Ontology?

Software usually begins by defining classes.

NATIA begins by defining concepts.

Classes evolve.

Concepts endure.

If the ontology is correct, architecture becomes a consequence.

---

# The Universe of NATIA

Everything inside NATIA belongs to one of six domains.

```
People

Knowledge

Work

Capabilities

Resources

Time
```

Every concept belongs to one—and only one—domain.

---

# Domain: People

People create purpose.

Only people possess intentions.

Only people validate knowledge.

Only people define success.

Concepts:

- Person
- Team
- Collaboration

---

# Domain: Knowledge

Knowledge is the permanent asset of every Initiative.

Knowledge survives sessions.

Knowledge survives technologies.

Knowledge survives models.

Concepts:

- Knowledge
- Context
- Decision
- Observation
- Experience
- Documentation
- Reference
- Relationship

---

# Domain: Work

Work transforms intentions into results.

Concepts:

- Initiative
- Objective
- Session
- Action
- Result
- Milestone

Work is always associated with a Workspace.

---

# Domain: Capabilities

Capabilities define what NATIA can do.

Capabilities are abstract.

They are independent from implementation.

Examples:

- Observe
- Search
- Remember
- Analyze
- Plan
- Execute
- Explain
- Generate
- Communicate
- Learn

Capabilities never describe technologies.

---

# Domain: Resources

Resources implement capabilities.

Resources are replaceable.

Examples include:

- Language Models
- Agents
- Tools
- Repositories
- Databases
- Documentation
- External Services
- APIs

Resources evolve continuously.

Capabilities remain stable.

---

# Domain: Time

Nothing inside NATIA is static.

Everything evolves.

Time is therefore a first-class concept.

Concepts include:

- Event
- History
- Timeline
- Version
- Session
- Evolution

Understanding an Initiative requires understanding its history.

---

# Fundamental Relationships

The ontology is governed by a small number of immutable relationships.

```
People create Intentions.

Intentions create Initiatives.

Initiatives generate Knowledge.

Knowledge creates Context.

Context enables Decisions.

Decisions trigger Actions.

Actions produce Results.

Results generate new Knowledge.

Knowledge evolves over Time.

Capabilities operate on Resources.

Resources assist People.
```

Everything inside NATIA can ultimately be described using these relationships.

---

# The Principle of Separation

NATIA distinguishes between:

What exists.

What happens.

Who performs it.

How it is implemented.

These concepts must never be confused.

For example:

A Capability is not an Agent.

An Agent is not a Model.

A Model is not Knowledge.

Knowledge is not Documentation.

Documentation is only one possible representation of Knowledge.

---

# The Principle of Identity

Every entity inside NATIA has an identity.

That identity remains stable over time.

Its state may evolve.

Its relationships may evolve.

Its implementation may evolve.

Its identity does not.

---

# The Principle of Evolution

Nothing inside NATIA is ever considered complete.

Everything evolves.

Knowledge evolves.

Initiatives evolve.

Capabilities evolve.

Even the ontology itself may evolve.

Evolution is not an exception.

It is the natural state of the system.

---

# Summary

NATIA is built upon concepts.

Architecture exists to implement those concepts.

Technology exists to support that architecture.

The ontology therefore becomes the most stable layer of the entire system.

Everything else is expected to evolve.
